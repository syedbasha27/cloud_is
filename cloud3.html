⚙️ Cloud Layers, Principles & Risks

System models, cloud computing layers, desired features, basic principles, risks and challenges

⏱ ~35 min 📊 15% weightage
🎯 Learning Objectives: Understand cloud system models, computing layers, desired features, basic principles, and identify cloud risks and challenges.
3.2 Cloud System Models

Cloud system models describe how cloud infrastructure is organized and accessed. Three main delivery models:

☁️ Public Cloud

Services available to the general public over the internet. Managed by CSP. Multi-tenant. Pay-per-use. Examples: AWS, Azure, Google Cloud.

🔒 Private Cloud

Exclusively for one organization. Highest security and control. Can be on-premise or hosted by vendor. Examples: OpenStack, VMware vSphere.

🔀 Hybrid Cloud

Combination of public and private clouds connected via LAN/WAN/VPN. Best of both worlds — flexibility + security.

3.3 Cloud Computing Layers
  • Hardware Layer: Physical servers, networking equipment, storage devices, power supply, cooling systems in data centers
  • Infrastructure Layer (IaaS): Virtualized resources — compute, storage, network. Hypervisors create VMs. Resource pooling.
  • Platform Layer (PaaS): OS, runtime environments, middleware, development frameworks, databases
  • Application Layer (SaaS): End-user applications delivered over the internet
3.4 Basic Principles of Cloud Computing
⭐⭐ These principles are exam favourites!
  • Scalability: Automatically add/remove resources based on demand
  • Elasticity: Dynamic resource provisioning — scale up/down seamlessly
  • Pay-per-use: Utility computing model — pay only for what is consumed
  • Reliability: Cloud infrastructure is designed for high availability and fault tolerance
  • Availability: Services accessible anytime, anywhere (broad network access)
  • Security: Multi-layer security approach to protect data and services
  • Virtualization: Core enabling principle — abstracts hardware to create virtual resources
  • Automation: Self-service, automated provisioning reduces human intervention
3.5 Desired Features of Cloud
  • Openness: Standards-based, interoperable APIs
  • Heterogeneity: Support diverse hardware and software platforms
  • QoS (Quality of Service): Performance guarantees via SLAs
  • SLA (Service Level Agreement): Contract defining expected service levels and uptime
  • Portability: Ability to move data/apps between cloud providers
  • Interoperability: Ability to work across different cloud platforms
3.6 Risks and Challenges in Cloud Computing
⭐⭐ Security is the #1 concern in cloud adoption!

Major Risks:

  • Data Security & Privacy: Data stored on shared infrastructure, risk of unauthorized access or breaches
  • Vendor Lock-in: Difficulty migrating between cloud providers due to proprietary formats
  • Downtime & Availability: Outages violate SLA. Reliability depends on CSP.
  • Compliance & Regulatory Issues: Data sovereignty laws may restrict where data can be stored
  • Limited Customization: Public cloud restricts deep customization
  • Network Latency: Inherent in WAN connectivity — all cloud apps suffer from latency
  • Multi-tenancy Risk: Data breaches by malicious co-tenants
  • Reduced visibility and control over data
⭐ Which service provides LEAST security? IaaS provides the least built-in security. SaaS provides the most. (Provider manages more at SaaS level.)

Cloud Security Alliance (CSA):

Organization that promotes cloud computing security standards. Key governance body for cloud security.

📋 Key Definitions
SLA (Service Level Agreement)

Contract between cloud provider and customer defining expected service levels, uptime guarantees (e.g., 99.9% availability), and penalties for violations. Downtime violates SLA.

⭐⭐⭐

QoS (Quality of Service)

Performance guarantees provided by cloud service. Includes metrics like latency, bandwidth, availability, and throughput. Introduced by SOC as fundamental concept for cloud.

⭐⭐

Elasticity

Dynamic provisioning and release of cloud resources based on demand. Unlike scalability (adding capacity), elasticity means seamless auto-scaling both up and down.

⭐⭐

Vendor Lock-in

Dependency on a single cloud provider's proprietary technology that makes it difficult or costly to migrate to another provider.

⭐⭐

Cloud Security Alliance (CSA)

Organization that promotes cloud computing security standards and best practices. Defines SECaaS categories and cloud security reference model.

⚠️ Risks Mnemonic — "DVCLNM"

Data Security | Vendor Lock-in | Compliance | Latency | Network downtime | Multi-tenancy risk

Phrase: "Don't Venture Cloud Lightly — Need Mindfulness!"

🛡️ Security by Service — "ISP Backwards"

IaaS = Insecure (most user responsibility)
PaaS = Partially secured
SaaS = Secured (most provider-managed)

📋 SLA = "Service Level Agreement"

Think of it as a promise from your cloud provider. If they break the promise (downtime), the SLA is violated! It's like a restaurant promising food in 30 min.
