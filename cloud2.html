🏗️ Cloud & Other Architectures

Layered cloud architecture, P2P, Client-Server, Grid Computing, Server Virtualization

⏱ ~40 min 📊 15% weightage
🎯 Learning Objectives: Understand cloud layered architecture (IaaS/PaaS/SaaS stack), compare P2P vs client-server, understand grid computing nodes, and grasp server virtualization.
2.2 Cloud Layered Architecture

Think of cloud architecture like a multi-storey building: ground floor (IaaS = infrastructure/foundation), middle floor (PaaS = platform/walls), top floor (SaaS = finished rooms/services).

  • IaaS (Bottom): Provides virtualized infrastructure — computing, storage, networking. Foundation layer.
  • PaaS (Middle): Provides platform for creating computing scenarios on top of IaaS. Includes OS and required services. PaaS = IaaS + custom software stack
  • SaaS (Top): Deployed services through cloud computing. Users run apps remotely.
  • dSaaS: Data-Storage-as-a-Service — storage service component
⭐ Key Insight: PaaS provides virtualized computing platform with guaranteed computing power, reserved bandwidth for storage and internet access.
2.3 Peer-to-Peer (P2P) Architecture
  • Each host is called a "peer"
  • Each peer acts as both client AND server
  • Fully decentralized — no central server
  • Easily scalable — just add more peers
  • Disadvantage: algorithm management is complex
  • Security concern: users allowed to access resources on their own
  • Can survive attacks well — no single point of failure
  • Works best with many active peers
  • Best for small workload chunks that can be reassembled
  • Examples: BitTorrent, Kazaa, Instant Messaging, VoIP
P2P vs Cloud differences: Cloud = easily scaled, hosted apps, highest security config. P2P = relatively inexpensive, simple to setup.
2.4 Client-Server Architecture
  • Two major components: Client and Server
  • Communication uses Application Layer protocols
  • Unidirectional: client sends request → server returns response
  • Many-to-one: multiple clients, one server
  • Nodes are independent → easy to replace/upgrade
  • Can work across different platforms
  • Disadvantage: Server overload if all clients request simultaneously → network congestion
  • High cost of maintenance and setup

Two Client Models:

  • Thin Client: Client = lightweight (only data retrieval). Server = heavy processing.
  • Fat Client: Client = does processing/transformation. Server = light (data access only).

Three Tiers:

  • Presentation layer (UI)
  • Application Logic layer (business rules)
  • Data Storage layer (database)

Examples: Mail servers, File servers (Google Docs), Web servers

2.5 Grid Computing
  • Group of networked computers forming a virtual high-performance compute engine
  • Subset of distributed computing
  • All machines work under same protocol
  • Used for: analyzing huge datasets, high-compute simulations

3 Types of Grid Nodes:

  • Control Node: Administrative node. Manages entire network. Schedules tasks. Uses Middleware software. Ensures no provider is overloaded.
  • Provider: Contributes resources to the network pool
  • User: End user consuming resources
⭐ Grid is NOT centralized — no servers required except the control node (which controls, not processes).

Grid supports heterogeneous machines with different OS. Tasks are divided into smaller parallel subtasks.

2.6 Server Virtualization
  • Masking server resources from server users
  • Hypervisor sits between hardware and VMs
  • Multiple VMs on one physical server
  • Benefits: better resource utilization, isolation, flexibility
  • Types: Full virtualization, Para-virtualization, OS-level virtualization
📋 Key Definitions
Peer (P2P)

A host in P2P architecture that acts as both client and server. It provides services to other peers and requests services from them simultaneously.

⭐⭐ Frequently tested

Thin Client

Client model where the client has lightweight implementation (data retrieval only) and the server handles all heavy data processing.

⭐⭐

Fat Client

Client model where the client processes and transforms data before returning to user. Server has relatively light implementation.

⭐⭐

Control Node (Grid)

Administrative node in grid computing. Manages the entire network, keeps accounting of resources, prioritizes and schedules tasks, uses Middleware for control.

⭐⭐⭐

Middleware

Software/networking protocol used in grid computing to administer the network. Control nodes are its executors.

⭐

dSaaS

Data-Storage-as-a-Service. Provides storage services in the cloud layered architecture.

⭐

🧠 Memory Tricks
🏗️ Cloud Layers — "IPS" Bottom to Top

IaaS (Infrastructure) → PaaS (Platform) → SaaS (Software)

Like a pizza: I (dough/base), P (sauce/platform), S (toppings/software you enjoy!)

🔗 P2P — "Everyone Equal"

Every peer is equal. No boss (no central server). Think of it as a circle of friends — anyone can ask anyone for help.

🖥 Client-Server — "Boss and Workers"

Server = boss (central, powerful). Clients = workers (request from boss). If too many workers ask the boss at once → boss gets overwhelmed (server overload).

⚡ Grid Nodes — "CPU"

Control node (manages) | Provider node (gives resources) | User node (uses resources)

👔 Thin vs Fat Client Trick

Thin = thin wallet (no resources on client side, server does work). Fat = fat wallet (client has resources to process data itself).

⚡ 5-Minute Revision
1. Cloud Layers (bottom→top): IaaS → PaaS → SaaS. PaaS = IaaS + custom software stack.
2. P2P: Each node = peer. Both client AND server. Decentralized. No central server. Resilient to attacks.
3. Client-Server: Unidirectional (client→server). Thin client = server processes. Fat client = client processes. Disadvantage: server overload.
4. Grid Nodes: Control (admin+scheduler) | Provider (gives resources) | User (consumes). Middleware controls. Heterogeneous machines supported.
5. Grid ≠ Centralized: No processing servers — only control node for management.
⚠️ Trap: Control node in grid MANAGES, it doesn't process tasks. Provider nodes contribute resources.
